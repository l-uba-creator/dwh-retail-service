# DWH для операционной аналитики розничного сервиса (ремонт и хранение техники)

Рабочие материалы к практическому примеру из отчёта по производственной практике
«Архитектура DWH для операционной аналитики розничного сервиса: моделирование
схемы и оптимизация запросов по истории ремонтов и хранения техники».

Репозиторий содержит только код и инфраструктуру примера — сам отчёт (docx) сюда
не выкладывается.

## Что здесь исправлено относительно приложений к отчёту

- **`sql/07_dq_status_order_check.sql`** — в исходном варианте (Приложение ПА.7)
  проверка порядка статусов вызывала `LAG()` прямо в `WHERE`, что в PostgreSQL не
  выполняется (оконные функции недоступны на этапе `WHERE`). Здесь `LAG` сначала
  считается в CTE, а фильтрация аномалий выполняется во внешнем запросе.
- **`dags/dwh_load_pipeline.py`** — в исходном варианте (Приложение ПА.5) задачи
  `ensure_partition` и `load` создавались в двух циклах без вызовов `>>`, порядок
  выполнения не гарантировался. Здесь задачи сгруппированы в `TaskGroup` и явно
  связаны: `ensure_partitions >> load_facts >> dq_status_order_check >>
  build_marts >> reconciliation`.
- Таблица замеров производительности отчёта заменена на **реальные цифры**
  (`results/table3_real.md`), полученные прогоном на этом стенде, а не оценочные.

## Стек и почему он отличается от отчёта

Отчёт описывает стек Docker Compose (Postgres + Airflow + Metabase). На машине,
где готовился этот пример, Docker Desktop не смог запуститься (нет WSL2/Hyper-V,
включение требует прав администратора и перезагрузки — решение оставлено на
усмотрение того, кто будет воспроизводить пример). Поэтому здесь:

- **PostgreSQL 16** — нативная установка Windows (или Docker — см. `docker-compose.yml`,
  если WSL2/Hyper-V доступны).
- **Metabase 0.50.x** — запускается как `metabase.jar` напрямую на Java 11+
  (без Docker), см. ниже.
- **Apache Airflow** — DAG (`dags/dwh_load_pipeline.py`) содержит исправленный,
  корректный код графа зависимостей (`TaskGroup` + явные `>>`). Полноценный запуск
  `airflow dags test` требует Linux-контейнер (Airflow официально не
  поддерживается на нативном Windows), который на этой машине недоступен без
  WSL2/Docker. Поэтому тот же граф задач (`ensure_partitions -> load_facts ->
  dq_status_order_check -> build_marts -> reconciliation`) реально выполнен и
  проверен через `scripts/run_pipeline.py` — включая двойной прогон, доказывающий
  идемпотентность (см. `results/idempotency_check.log`). Если Docker/WSL2
  доступны, DAG можно запустить как есть: см. команду в конце этого файла.

## Структура

```
sql/            DDL staging/DDS/marts, идемпотентная загрузка, DQ-проверки, реконсиляция
generator/      генератор синтетических данных (уменьшенный, но реалистичный объём)
dags/           DAG Apache Airflow (исправленные зависимости)
scripts/        run_pipeline.py (полный цикл + проверка идемпотентности),
                benchmark.py (честные замеры "сырые/индексы/партиции/витрины"),
                setup_metabase.py (headless-настройка дашборда через REST API)
results/        реальные результаты прогонов (замеры, логи, скриншоты дашборда)
docker-compose.yml   опциональный вариант через Docker (Postgres + Metabase)
```

## Как воспроизвести (нативно, без Docker)

1. Поднимите PostgreSQL 16, создайте пользователя и базу для проекта:
   ```sql
   CREATE ROLE dwh_practice WITH LOGIN PASSWORD '...';
   CREATE DATABASE dwh_practice OWNER dwh_practice;
   ```
2. Сгенерируйте данные:
   ```
   python generator/generate_data.py
   ```
3. Задайте переменные окружения подключения (PGHOST, PGPORT, PGDATABASE, PGUSER,
   PGPASSWORD, PSQL_PATH) и запустите полный цикл:
   ```
   python scripts/run_pipeline.py full
   ```
4. Проверка идемпотентности (повторный прогон не должен менять число строк):
   ```
   python scripts/run_pipeline.py idempotency-check
   ```
5. Замеры производительности (реальные, не оценочные):
   ```
   python scripts/benchmark.py
   ```
   Результат — `results/table3_real.md` и сырые планы `EXPLAIN ANALYZE` в
   `results/benchmark_raw/`.
6. Metabase:
   ```
   java -jar .local/metabase.jar
   ```
   (см. `.local/README` — jar не хранится в репозитории), затем:
   ```
   python scripts/setup_metabase.py
   ```
   Скрипт настраивает подключение к БД, создаёт карточки и дашборд, включает
   Public Sharing, пишет ссылку в `results/metabase_dashboard_info.json`.
7. Проверка DAG (нужен Docker для контейнера Airflow):
   ```
   docker run --rm --network dwh_net \
     -e AIRFLOW_CONN_POSTGRES_DEFAULT=postgresql://dwh_practice:...@postgres:5432/dwh_practice \
     -v ./dags:/opt/airflow/dags -v ./sql:/opt/airflow/sql \
     apache/airflow:2.9.3 bash -c "airflow db init && airflow dags test dwh_load_pipeline 2025-06-01"
   ```

## Данные

Синтетические данные сгенерированы `generator/generate_data.py`. Масштаб
уменьшен относительно исходной оценки отчёта (10 млн/4 млн строк) до объёма,
воспроизводимого на обычном ноутбуке за разумное время (см.
`results/generation_summary.json` — точные числа конкретного прогона). В данные
намеренно подмешан брак (недопустимые ссылки, нарушенный порядок статусов) —
чтобы проверить работу карантина и DQ-контроля на реальных, а не гипотетических
случаях.

## Результаты реального прогона на этом стенде

- Сгенерировано: 300 000 заявок в ремонт (1 560 133 события), 150 000 потоков
  хранения (525 304 события) за 12 месяцев 2025 года, включая намеренно
  испорченные записи (631 заявка с недопустимой ссылкой, 863 заявки с
  нарушенным порядком статусов).
- Загружено в `dds.fact_repair`: 1 556 809 строк; в `dds.fact_storage`: 525 255
  строк; в карантин `dds.fact_repair_quarantine`: 4 673 строки
  (unresolved-reference + status-order-anomaly).
- Построены витрины: `mart_repair_sla` — 43 156 строк, `mart_center_load` —
  5 462, `mart_storage_turnover` — 334 221.
- Реконсиляция (`sql/08_reconciliation_check.sql`) прошла без расхождений: сумма
  `tickets_total` по витрине SLA совпала с числом завершённых заявок в
  `fact_repair`.
- Идемпотентность подтверждена двойным прогоном на одном и том же окне данных —
  число строк идентично между прогонами (`results/idempotency_check.log`).
- Реальные замеры производительности (`results/table3_real.md`): Q1 (SLA/среднее
  время ремонта) ускорился в **171.6×** при переходе от сырых фактов к витринам
  на уменьшенном стенде — направление эффекта то же, что и в исходной оценке
  отчёта (60–80× на 10 млн строк), но абсолютные цифры честно пересчитаны под
  реальный объём и реальное железо.
- Дашборд Metabase поднят, подключён к витринам, дашборд «Операционная
  аналитика розничного сервиса» с 5 карточками собран и опубликован через
  Public Sharing; скриншот — `results/metabase_dashboard_1.png`.

## Известное ограничение генератора

При старте месячного окна `2025-12` staging-загрузка складских событий даёт на
49 строк меньше, чем сгенерировано (525 304 сгенерировано → 525 255 загружено,
~0.01%) — часть событий одного потока хранения попадает на границу между
месячными окнами загрузки. На демонстрационном масштабе это не влияет на
качество витрин, но при продовом использовании такого генератора стоит либо
грузить каждый поток хранения целиком в одно окно, либо расширять окно на
сутки в обе стороны.
